# Техническое задание: Telegram-бот для текстовой игры

## Общая информация

Игра представляет собой текстовое интерактивное приключение, полностью проходящее через Telegram-бота.

Срок работы проекта ограничен:

- Игра работает в течение 3 дней.

- После завершения мероприятия бот может быть отключён.

- Ожидаемое количество игроков: менее 30 одновременно.

---

# Основные сущности

## Акт

Контент разделён на 3 акта.

Порядок прохождения:

```text
Акт 1 → Акт 2 → Акт 3
```

Особенности:

- Акт 2 становится доступным только после завершения Акта 1.

- Акт 3 становится доступным только после завершения Акта 2.

- После открытия нового акта предыдущие акты остаются доступными.

- Возможна реализация механики одновременной блокировки всех актов после выполнения определённых условий (не определено окончательно).

По завершении каждого акта бот должен отправлять назначенный сегмент завершения акта.

---

## Сегмент

Сегмент является основной единицей контента.

Каждый сегмент имеет уникальный идентификатор.

Сегмент может содержать:

- текст;

- одно или несколько медиа;

- окно взаимодействия с пользователем;

- внутренние действия;

- переходы в другие сегменты.

Допускается:

- несколько сегментов могут вести в один и тот же сегмент;

- один сегмент может содержать несколько медиафайлов;

- один сегмент может выполнять несколько действий одновременно.

Пример действий:

- отправить сообщение;

- изменить переменную;

- запустить таймер;

- открыть другой сегмент;

- открыть акт.

---

## Скрытые сегменты

Не являются обязательными для текущей версии проекта.

Архитектура должна допускать наличие скрытых сегментов, которые могут запускаться:

- по условию;

- по таймеру;

- по событию.

---

# Система взаимодействия

## Выбор варианта ответа

Бот должен поддерживать выбор одного варианта из нескольких предложенных.

Максимальное количество вариантов ответа:

```text
4
```

Тип реализации:

Не определён.

Возможные варианты:

- Inline Keyboard;

- Reply Keyboard.

---

## Ввод текста

Бот должен поддерживать ввод текста пользователем.

Для каждого текстового задания задаётся список правильных ответов.

Пример:

```text
пароль
секретный пароль
password
```

Особенности:

- допускается несколько правильных вариантов;

- регистр игнорируется;

- допускается нормализация пробелов;

- допускается нормализация пунктуации.

Если введён неправильный ответ:

- ответ считается неверным;

- выполняется назначенное действие;

- текущий сегмент повторяется.

Если введён неожиданный тип данных:

- ввод отклоняется;

- пользователю отправляется соответствующее сообщение.

---

# Система переменных

Поддерживаются следующие типы переменных:

- Integer

- Boolean

- String

Поддерживаются два уровня хранения:

## Переменные пользователя

Примеры:

```text
current_act
current_segment
trust
uploaded_file
```

## Глобальные переменные

Примеры:

```text
event_started
event_finished
```

Переменные могут:

- участвовать в условиях;

- изменяться сегментами;

- использоваться для открытия контента.

---

# Система условий

Сегменты могут иметь условия входа.

Пример:

```text
trust >= 10
and
timer_completed == true
```

При выполнении условий сегмент становится доступным.

---

# Система таймеров

Бот должен поддерживать внутренние таймеры.

Таймеры используются для:

- запуска событий;

- открытия контента;

- изменения переменных;

- переходов между сегментами.

Особенности:

- точность до минут;

- у пользователя может существовать несколько активных таймеров одновременно;

- по завершении таймера выполняется назначенное событие.

Для событий должна поддерживаться система приоритетов.

Если несколько событий должны выполниться одновременно:

- сначала выполняется событие с более высоким приоритетом.

---

# Медиафайлы

## Отправка медиа ботом

Сегмент может содержать один или несколько медиафайлов.

---

## Получение медиа от пользователя

Поддерживаемые форматы:

- PNG (обязательно)

- JPEG/JPG (желательно)

Все остальные форматы:

- отклоняются;

- пользователю повторно отправляется текущий сегмент.

Ограничение размера файла:

```text
10 МБ
```

Размер должен быть настраиваемым.

Особенности:

- один пользователь может загрузить только один файл;

- файл сохраняется;

- оригинальный файл сохраняется без изменений;

- файл может использоваться в дальнейшей логике игры.

---

# Сообщения

Бот должен поддерживать:

- отправку сообщений;

- удаление собственных сообщений;

- удаление сообщений пользователя.

Удаление сообщений выполняется с задержкой.

Величина задержки определяется настройками.

---

# Имитация живого общения

Бот должен поддерживать искусственную задержку перед ответом.

Требования:

- отображение статуса "печатает...";

- задержка рассчитывается автоматически на основании длины сообщения;

- коэффициенты задержки должны быть настраиваемыми.

---

# Система логирования

Необходимо логировать:

- начало игры;

- переходы между сегментами;

- выборы пользователя;

- правильные ответы;

- неправильные ответы;

- запуск таймеров;

- завершение таймеров;

- загрузку файлов;

- завершение актов;

- административные действия.

---

# Административная система

Все администраторы обладают полными правами.

Минимальный набор возможностей:

- просмотр состояния игрока;

- просмотр текущего акта;

- просмотр текущего сегмента;

- просмотр переменных;

- просмотр таймеров;

- просмотр загруженных файлов;

- изменение переменных;

- сброс прогресса игрока;

- открытие актов;

- отправка глобальных сообщений;

- остановка игры;

- запуск игры.

---

# Контент

Контент должен быть реализован через внешние данные.

Жёсткое кодирование контента в коде приложения не рекомендуется.

Предпочтительно:

- JSON;

- YAML;

- иной аналогичный формат.

Окончательное решение остаётся за разработчиком.

---

# Обновление контента

При изменении сегмента после запуска проекта игрок должен получать новую версию сегмента.

---

# Аналитика

Необходимо собирать:

- количество игроков;

- прохождение актов;

- прохождение сегментов;

- выборы игроков;

- количество неправильных ответов;

- загруженные файлы;

- среднее время прохождения.

---

# Архитектурные рекомендации

Рекомендуется реализация в виде конечного автомата (State Machine).

Пример структуры:

```text
Акт
 └─ Сегмент
      ├─ Контент
      ├─ Условия
      ├─ Действия
      ├─ Таймеры
      └─ Переходы
```

Текущее ТЗ не требует обязательного использования данного подхода, однако архитектура должна поддерживать дальнейшее расширение проекта.
