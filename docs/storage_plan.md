# Хранение данных и файлов

## Принятое решение

Для проекта используем Supabase:

- Postgres хранит игроков, прогресс, переменные, события, таймеры и метаданные загрузок.
- Supabase Storage хранит пользовательские изображения.
- В базе не хранятся сами изображения, только путь к объекту в Storage.

## Bucket для загрузок

Рекомендуемый bucket:

```text
player-uploads
```

Настройки:

- тип: private;
- максимальный размер файла: 10 МБ;
- допустимые типы: PNG и JPEG/JPG.

Бот работает с bucket через `SUPABASE_SERVICE_ROLE_KEY`, поэтому этот ключ нельзя отдавать клиентам, публиковать в репозитории или вставлять в frontend.

## Расчет объема

Если 100 игроков загрузят по одному файлу 10 МБ:

```text
100 * 10 МБ = примерно 1 ГБ
```

Это близко к лимиту Supabase Free Storage, поэтому для реального события лучше иметь запас: либо заранее сжимать изображения, либо временно перейти на платный план Supabase.

## Что записываем в базу

`players`:

- Telegram ID;
- имя и username;
- текущий акт;
- текущий сегмент;
- время старта и последнего обновления.

`player_state`:

- игровые переменные в JSON.

`events`:

- старт игры;
- открытие сегмента;
- выбранные варианты;
- правильные и неправильные ответы;
- загрузки;
- ошибки;
- будущие административные действия.

`uploads`:

- Telegram file_id;
- путь в Supabase Storage;
- размер;
- mime-type;
- сегмент, в котором файл был загружен.

`timers`:

- время срабатывания;
- приоритет;
- действие в JSON;
- статус.

## Правило для зина

Если нужен максимум качества, просим игрока отправлять изображение как файл. Если отправить как обычное фото, Telegram может сжать изображение.
